## TIER 1 -- PLAY STORE BLOCKERS

### 1. TWA Bubblewrap project does not exist

- **What is wrong:** No `twa-android/` folder. Cannot generate an APK/AAB without it.
- **What needs to happen:** Run `bubblewrap init` with the correct `twa-manifest.json` config pointing to `https://cmgroups.vercel.app`, or create the project manually. This generates the full Android project.
- **Files to change:** New folder `twa-android/` with Bubblewrap output
- **Can Cursor do this automatically:** Partially -- Cursor can generate a `twa-manifest.json` file. The actual `bubblewrap init` and `bubblewrap build` commands require Node.js + JDK + Android SDK installed locally, plus interactive prompts.
- **If manual:** Owner needs to install `@nickersoft/bubblewrap-cli` (or `@nickersoft/nickersoft/nickersoft` -- the official `@nickersoft/nickersoft` is now `@nickersoft/nickersoft`), JDK 11+, and Android SDK. Then run `bubblewrap init --manifest https://cmgroups.vercel.app/manifest.webmanifest`.
- **Effort:** medium

### 2. assetlinks.json has placeholder fingerprints

- **What is wrong:** Both SHA-256 fingerprints are `TODO:REPLACE_WITH_...`. Without real fingerprints, TWA verification fails and Chrome shows the URL bar (defeats the purpose).
- **What needs to happen:** After generating a signing key (during `bubblewrap init`), extract the SHA-256 fingerprint and update the file. After uploading to Play Console, also add the Play App Signing fingerprint.
- **Files to change:** [frontend/public/.well-known/assetlinks.json](frontend/public/.well-known/assetlinks.json)
- **Can Cursor do this automatically:** No -- requires the actual keystore fingerprint.
- **If manual:** Run `keytool -list -v -keystore <keystore-file>` and copy the SHA-256 fingerprint. After Play Console enrollment, copy the second fingerprint from Setup > App signing.
- **Effort:** low

### 3. Frontend VAPID public key missing

- **What is wrong:** `VITE_VAPID_PUBLIC_KEY` is not defined in `frontend/.env`. The `usePushNotifications` hook reads it via `import.meta.env.VITE_VAPID_PUBLIC_KEY`. Without it, `isSupported` is always `false` and push notifications silently fail.
- **What needs to happen:** Copy the `VAPID_PUBLIC_KEY` value from `backend/.env` and add it to `frontend/.env` as `VITE_VAPID_PUBLIC_KEY=<value>`. Also add it to Vercel environment variables.
- **Files to change:** [frontend/.env](frontend/.env), Vercel dashboard env vars
- **Can Cursor do this automatically:** Yes (Cursor can add the env var line). The Vercel dashboard update is manual.
- **Effort:** low

### 4. Feature graphic (1024x500) missing

- **What is wrong:** Play Store requires a feature graphic. No such file exists.
- **What needs to happen:** Create a 1024x500 PNG promotional graphic showing the app brand/value prop. Place it at `frontend/public/feature-graphic.png` for reference.
- **Can Cursor do this automatically:** Cursor can generate a placeholder SVG. The real graphic needs design work.
- **If manual:** Owner or designer needs to create a 1024x500 branded image.
- **Effort:** low

### 5. Real app screenshots needed

- **What is wrong:** The 2 screenshot files (`home.png`, `products.png`) are generated placeholders from the icon generation script (solid color with text), not actual app screenshots.
- **What needs to happen:** Take real screenshots of the running app (1080x1920 for phones, optionally 2560x1600 for tablets). Play Store requires at least 2, recommends 4-8.
- **Can Cursor do this automatically:** No -- requires a running app.
- **If manual:** Open `https://cmgroups.vercel.app` on a phone or emulator, take screenshots of Home, Products, Services, Courses pages.
- **Effort:** low

### 6. Real app icon / logo needed

- **What is wrong:** All icon PNGs are generated from a simple SVG placeholder (`icon.svg` = pink rectangle with "CM GROUPS" text). Play Store reviewers may reject or the app will look unprofessional.
- **What needs to happen:** Replace `frontend/public/icon.svg` (or provide `icon-source.png`) with the real brand logo, then re-run `npm run generate-icons`.
- **Can Cursor do this automatically:** No -- requires the real logo file.
- **If manual:** Provide a 1024x1024 PNG logo file, place at `frontend/public/icon-source.png`, run `npm run generate-icons`.
- **Effort:** low

## TIER 2 -- PWA QUALITY

### 7. Offline fallback not wired in service worker

- **What is wrong:** `offline.html` exists and is included in `includeAssets`, but the SW's `NavigationRoute` uses `NetworkFirst` with no catch handler. If the network is down and the page isn't cached, the user sees a browser error instead of `offline.html`.
- **What needs to happen:** Add a `setCatchHandler` in `src/sw.js` that serves `/offline.html` for failed navigation requests.
- **Files to change:** [frontend/src/sw.js](frontend/src/sw.js)
- **Can Cursor do this automatically:** Yes
- **Effort:** low

### 8. No `wide` form factor screenshot in manifest

- **What is wrong:** Both screenshots are `narrow` (phone). Chrome desktop install prompts benefit from a `wide` screenshot. Not a blocker but improves desktop PWA install experience.
- **What needs to happen:** Add at least one screenshot with `"form_factor": "wide"` (e.g., 1920x1080).
- **Files to change:** [frontend/vite.config.js](frontend/vite.config.js) (manifest.screenshots array)
- **Can Cursor do this automatically:** Yes (add the config entry). The actual screenshot file needs to be provided.
- **Effort:** low

### 9. Redundant `push-sw.js` in public/

- **What is wrong:** `frontend/public/push-sw.js` duplicates the push/notificationclick handlers already in `src/sw.js`. It's unused (no code registers it) and may confuse future maintainers.
- **What needs to happen:** Delete `frontend/public/push-sw.js`.
- **Files to change:** Delete [frontend/public/push-sw.js](frontend/public/push-sw.js)
- **Can Cursor do this automatically:** Yes
- **Effort:** low
